# InjectedToolArg in LangChain – Why It Is Useful

## 1. What Problem Does Injection Solve?

In many LLM tool workflows, **one tool depends on the output of another tool**.

Example in this project:

1. First tool → fetch currency conversion rate
2. Second tool → convert currency using that rate

The problem is that **the LLM does not know the conversion rate beforehand**, because it must be fetched from an external API.

If the tool is defined normally, the LLM must generate all arguments itself.

Example:

```
convert(
    base_currency_value = 10,
    conversion_rate = 92.7
)
```

But the LLM **cannot know the exact conversion rate**, so it might:

* Guess the value
* Hallucinate the value
* Produce incorrect calculations

This leads to unreliable tool execution.

---

# 2. Solution: InjectedToolArg

LangChain provides **InjectedToolArg** to solve this problem.

```
conversion_rate: Annotated[float, InjectedToolArg]
```

This tells LangChain:

> "This parameter should NOT be generated by the LLM.
> It will be injected by the system during runtime."

So the LLM only provides the arguments it actually knows.

Example tool call generated by the LLM:

```
{
  "name": "convert",
  "args": {
    "base_currency_value": 10
  }
}
```

The system then **injects the conversion rate automatically**.

Final execution becomes:

```
convert(10, 92.7)
```

---

# 3. Workflow Comparison

## Without Injection

```
User Prompt
      ↓
LLM generates tool call
      ↓
convert(10, ???)
```

The LLM must guess the conversion rate.

Problems:

* Hallucinated values
* Incorrect calculations
* Unreliable tool behavior

---

## With InjectedToolArg

```
User Prompt
      ↓
LLM decides first tool
      ↓
get_conversion_factor(USD, INR)
      ↓
API returns conversion rate
      ↓
convert(10)
      ↓
LangChain injects conversion_rate
      ↓
convert(10, 92.7)
```

The LLM does **not guess the value**, and the system provides the real value.

---

# 4. Benefits of InjectedToolArg

| Feature                | Benefit                                             |
| ---------------------- | --------------------------------------------------- |
| Prevents hallucination | LLM does not guess unknown values                   |
| Secure tool pipelines  | Sensitive values are hidden from the LLM            |
| Enables tool chaining  | Output of one tool can feed another tool            |
| Cleaner tool design    | Tools clearly separate LLM inputs and system inputs |
| More reliable results  | Runtime values come from real sources               |

---

# 5. Real-World Use Cases

InjectedToolArg is commonly used for:

### API Keys

```
search_api(query, api_key = InjectedToolArg)
```

The LLM never sees the API key.

---

### Database Connections

```
query_database(sql, db_connection = InjectedToolArg)
```

The LLM cannot access the database directly.

---

### Retrieval-Augmented Generation (RAG)

```
answer_question(
    question,
    retrieved_documents = InjectedToolArg
)
```

The retrieved documents come from the retriever, not the LLM.

---

# 6. Example from This Project

Tool 1:

```
get_conversion_factor("USD", "INR")
```

Output:

```
92.7
```

Tool 2:

```
convert(
    base_currency_value = 10,
    conversion_rate = InjectedToolArg
)
```

Final execution:

```
convert(10, 92.7)
```

Output:

```
927 INR
```

---

# 7. Key Takeaway

InjectedToolArg allows **runtime values to be safely passed into tools without requiring the LLM to generate them**.

This makes multi-step tool pipelines:

* safer
* more accurate
* easier to maintain

---

# Summary

InjectedToolArg helps LangChain build reliable tool workflows by separating:

* **LLM-generated inputs** (what the model knows)
* **system-injected inputs** (runtime values from other tools or systems)

This prevents hallucination, enables tool chaining, and improves the reliability of LLM applications.
